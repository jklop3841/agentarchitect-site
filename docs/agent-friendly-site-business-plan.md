# Agent-Friendly Website 商业计划 v1

日期：2026-04-07

## 1. 一句话定义

这不是普通个人官网，而是一个双层网站：

- 表层是 `智能体架构师卢成` 的个人品牌与能力名片
- 底层是 Agent 可发现、可调用、可验证的能力接口站

第一版不是卖“神秘加密”，而是卖一件更容易被 Agent 和开发者理解的事：

**让 Agent 在不获取内部 prompt、规则和方法论的前提下，稳定调用你的私有 workflow。**

---

## 2. 核心判断

### 为什么值得做

未来越来越多的流量不是“人点开网页”，而是：

- Agent 先发现你
- Agent 先读取你
- Agent 先比较你
- Agent 再决定是否路由到你

因此，未来的网站不只是给人看，还要给 Agent：

- 看得懂
- 调得起
- 用得放心
- 愿意持续回来

### 这个项目真正卖的不是秘密

不是卖“我有一个谁也看不懂的 prompt”。

真正卖的是：

- 比自己做更省事
- 比自己做更稳
- 比自己做更省上下文和试错成本
- 调用后拿不到你的内部方法论

---

## 3. 价值公式

一个 Agent 或开发者愿意持续调用你，通常要同时满足四件事：

### 1) 低摩擦

- 3 分钟内能理解并接入
- 文档短
- 示例能跑
- 默认值合理
- 不需要复杂销售或人工培训

### 2) 高价值

- 结果更稳
- 更快
- 更省上下文
- 出错更少
- 或者只有你这里能提供

### 3) 可负担

- 价格不是越低越好，而是相对替代方案更划算
- 让调用方觉得“比自己折腾便宜”

### 4) 可依赖

- 输入边界清晰
- 输出结构稳定
- 失败有明确返回
- 调用可验证
- 权限边界清楚

### 结论

真正的价值公式是：

`采用你 = 低摩擦 × 高价值 × 可负担 × 可依赖`

任何一项太弱，这个站都很难形成持续调用。

---

## 4. 目标用户

第一版只服务一类人，不要贪多。

### 主目标用户

Agent 开发者、自动化开发者、技术型独立开发者、小团队操盘手

他们最关心的是：

- 有没有现成能力可以接
- 能不能节省时间
- 会不会不稳定
- 成本会不会炸
- 能不能快速验证价值

### 不优先服务的人

- 普通消费者
- 只想看内容、不想接 API 的用户
- 需要长销售链条的大客户

---

## 5. 第一版产品定义

### 产品名

建议内部定义为：

`Private Workflow Runtime for Agents`

对外中文表达可用：

`给 Agent 调用的私有工作流运行时`

### 一句话卖点

**Agent 把输入发给你，你在服务端运行私有 workflow，返回结果；Agent 拿不到内部 prompt、规则和方法论。**

### 第一版只做 3 个能力

#### 1. `catalog`

告诉 Agent：

- 你有什么工具
- 每个工具做什么
- 输入输出是什么
- 调用限制是什么

#### 2. `run`

Agent 调用某个私有能力，收到结果。

#### 3. `verify`

验证某次结果是不是由你的服务生成，提升可信度。

### 第一版不做

- 不做开放式工作流上传平台
- 不做复杂多租户系统
- 不做“绝对不可破解”的宣传
- 不做模型基础设施
- 不做一次性铺太多能力

---

## 6. 为什么第一版不该主打“加密”

“加密”这个词在市场传播上有三个问题：

- 容易被理解成包装 prompt
- 容易被质疑安全承诺过大
- 太偏技术概念，不够结果导向

更好的说法是：

- controlled execution
- private workflow runtime
- sealed workflow access

这些词的重点不是“你看不到”，而是“你用得到，而且用得更省心”。

---

## 7. 商业模式

### 第一阶段目标

先证明有人持续调用，而不是先证明自动收款顺畅。

### 起步阶段建议

#### 免费层

- 少量免费调用
- 用于体验和传播
- 不承诺高 SLA

#### 开发者层

- 低价月付
- 提供稳定额度
- 适合个人开发者和小团队

#### 定制层

- 人工发 key
- 场景定制
- 可按结果或配额谈

### 价格原则

价格不是围绕“你成本多少”定，而是围绕“替代方案成本”定。

用户对比的是：

- 自己做一套 workflow 的时间
- 自己维护 prompt 和防泄漏的成本
- 自己 debug 的成本
- 自己承担不稳定输出的成本

### 支付问题的现实建议

如果短期没有 Stripe / PayPal 企业能力：

- 先不强求自动支付
- 先做人肉发 key
- 先做免费试用和种子用户
- 先积累调用案例
- 后续再补自动收费或虚拟币通道

现阶段最重要的不是支付，而是证明：

**有人会持续调用。**

---

## 8. 成本结构

### 核心成本分两层

#### 1. 发现成本

包括：

- 官网页面
- 文档
- `agents.txt`
- `agent.json`
- `openapi.json`
- 示例代码

这部分应该尽量做成静态内容，成本很低，主要是 CDN 和带宽。

#### 2. 执行成本

包括：

- `run`
- `verify`
- 日志
- 计量
- 鉴权
- 可能的模型调用

这部分才是变量成本。

### 低成本原则

- 被爬取几乎不消耗 token
- 真正执行才消耗 token
- 重复请求尽量缓存
- 优先规则与程序逻辑，少走长上下文模型
- 能用小模型不用大模型
- 能同步返回结构化结果就不要每次长文本生成

### 成本判断

如果结构设计合理，早期这不会是高成本项目。

真正会把账单打爆的，不是“Agent 很多”，而是：

- 所有访问都打进模型层
- 匿名无限调用
- 不做缓存和限流
- 输出过长

所以重点不是怕 Agent 多，而是要把架构分层。

---

## 9. 护城河判断

### 弱护城河

- 只是做个人官网
- 只是做 API 壳子
- 只是做一个概念站

这些都很容易被抄。

### 中等护城河

- 你的 workflow 质量更高
- 你的输入输出协议更稳定
- 你的文档和接入体验更好
- 你的可验证机制更成熟

### 更强护城河

- 你掌握别人没有的数据、流程、结构化规则
- 你形成“调用习惯”
- 你形成“被 Agent 默认信任的能力节点”

所以真正的护城河不是“我藏起来了”，而是：

**复制我的结果链路不划算。**

---

## 10. 最大风险

### 风险 1：概念先进，但没有高频调用

如果你讲的是未来，但今天没人刚需，网站会变成观念展示页。

### 风险 2：主打“加密”导致外界误解

会被理解成：

- prompt 套壳
- 安全营销
- 黑盒包装

### 风险 3：功能太散

如果同时做：

- 个人品牌
- 产品平台
- 多工具市场
- 商业收费
- 国际支付

第一版很容易失焦。

### 风险 4：没有可信交付机制

如果输出不稳定、失败不透明、权限不清，Agent 不会长期依赖你。

---

## 11. 第一版成功标准

第一版不看营收，先看这几个指标：

- 有明确的一句话产品定义
- 有 1-3 个真实可调用能力
- 有 Agent 可以读懂的机器入口
- 有 3 分钟内能跑通的接入体验
- 有至少几位种子用户完成重复调用
- 有稳定日志和调用计量

如果这些没做到，就不要急着谈大规模商业化。

---

## 12. 回到网站：这个网站到底该长什么样

这是最重要的一节。网站不是商业计划的附属，而是产品入口本身。

### 网站定位

这个网站要同时承担 4 个角色：

#### 1. 人类理解入口

让人类在 10 秒内知道：

- 你是谁
- 你做什么
- 为什么值得信任

#### 2. Agent 发现入口

让 Agent 能快速发现：

- 你有哪些能力
- 如何调用
- 是否有权限限制

#### 3. 技术接入入口

让开发者能直接复制示例完成调用。

#### 4. 商业转化入口

让真正想深入使用的人留下联系方式、申请 key、后续转付费。

### 第一版网站结构

#### 首页 `/`

只回答 4 个问题：

- 我是谁
- 我提供什么
- Agent 怎么接
- 为什么调我比自己做更值

#### 产品页 `/products/private-workflow-runtime`

详细解释：

- 适合谁
- 解决什么问题
- 输入输出
- 调用示例
- 为什么拿不到内部方法论

#### 文档页 `/docs`

按最短路径组织：

- 快速开始
- 鉴权
- `catalog`
- `run`
- `verify`
- 返回结构
- 错误处理

#### 机器入口

- `/agents.txt`
- `/.well-known/agent.json`
- `/openapi.json`
- `/api/catalog`
- `/mcp`

### 首页 Hero 推荐写法

不要先讲“未来趋势”。

先讲结果：

**Give agents access to private workflows without exposing the workflow itself.**

中文可对应：

**让 Agent 调用你的私有工作流，但拿不到你的内部方法论。**

副标题再讲你是谁和为什么值得信任。

### 网站的第一目标

不是“看起来高级”，而是让下面这句成立：

**一个 Agent 开发者在短时间内明白你是什么、怎么接、值不值接。**

---

## 13. 实施顺序

### 阶段 1：网站与机器入口

- 首页
- 产品页
- 文档页
- `agents.txt`
- `agent.json`
- `openapi.json`

### 阶段 2：最小可调用能力

- `catalog`
- `run`
- `verify`
- API key
- 限流
- 调用日志

### 阶段 3：种子用户验证

- 手动发 key
- 观察调用模式
- 看看哪些能力最常被调用

### 阶段 4：再决定收费和扩展

- 是否自动支付
- 是否加更多工具
- 是否做多层套餐

---

## 14. 最终结论

这个项目真正应该做成的，不是一个“讲 Agent 愿景”的网站，而是一个：

**以个人品牌为门面、以 Agent 调用为底座、以私有 workflow 运行时为首发产品的能力站。**

如果第一版能做到下面这件事，就值得继续投：

**让外部 Agent 以低摩擦、低成本、可验证的方式，稳定调用你的独有能力。**

做不到这件事，再漂亮的网站都只是包装。
